# Single-Cycle RISC-V CPU in Verilog

A complete single-cycle RV32I processor built from scratch in Verilog, block by block, over 9 days (Day 51–59) as part of the **#100DaysOfVLSI** challenge.

Every module was individually designed, simulated, and verified with GTKWave before being integrated into the final datapath.

---

## Overview

This project implements a single-cycle RISC-V CPU supporting a subset of the RV32I instruction set, including R-type arithmetic/logic instructions, I-type immediate instructions, loads, stores, and branches.

The design follows the classic 5-stage datapath model — Fetch, Decode, Execute, Memory, Writeback — collapsed into a single clock cycle per instruction.

---

## Architecture

```
   FETCH          DECODE           EXECUTE         MEMORY        WRITEBACK
┌─────────┐    ┌───────────┐    ┌──────────┐    ┌──────────┐   ┌─────────┐
│   PC    │───▶│  Decode   │───▶│   ALU    │───▶│   Data   │──▶│  Mux ───┼──▶ Reg File
│ + IMEM  │    │  + Ctrl   │    │ + ALU    │    │  Memory  │   │ (WB)    │
└─────────┘    │  + RegFile│    │  Ctrl    │    └──────────┘   └─────────┘
               └───────────┘    └──────────┘
```

Control signals (`reg_write`, `alu_src`, `mem_read`, `mem_write`, `mem_to_reg`, `branch`) are generated by the Control Unit and directed to every stage of the datapath.

## Module Breakdown

| Day | Module | Description |
|-----|--------|-------------|
| 51 | `alu.v` | 32-bit ALU — ADD, SUB, AND, OR, XOR, SLT, SRL, SRA |
| 52 | `regfile.v` | 32×32 register file, two read ports, one write port, x0 hardwired to zero |
| 53 | `pc_imem.v` | Program counter + instruction memory (ROM), word-aligned, +4 per cycle |
| 54 | `decode_unit.v` | Instruction field extraction + immediate generation (I, S, B, U, J types) |
| 55 | `control_unit.v` | Main control — generates high-level control signals from opcode |
| 56 | `alu_control.v` | ALU control — decodes funct3/funct7 into 3-bit ALU operation select |
| 57 | `data_mem.v` | Data memory — synchronous write, asynchronous read |
| 58 | — | Datapath integration design (architecture document, no new code) |
| 59 | `riscv_cpu.v` + `riscv_cpu_tb.v` | Full single-cycle CPU integrated — all blocks connected, verified end-to-end |

## Simulation Results

Test program executed:

```assembly
addi x1, x0, 5      # x1 = 5
addi x2, x0, 3       # x2 = 3
add  x3, x1, x2      # x3 = 8
```

**Waveform verified:**
- PC increments by 4 every clock cycle, no stalls, no X states
- ALU results match expected values for every instruction
- Register file correctly stores and forwards values
- Clean execution with zero unknown/undefined signals

GTKWave screenshots for every module are included in `/waveforms`.

## How to Run

```bash
# Compile
iverilog -o cpu_sim riscv_cpu.v riscv_cpu_tb.v

# Simulate
vvp cpu_sim

# View waveform
gtkwave riscv_cpu.vcd
```

## What's Next

This single-cycle design is the foundation for further work:

- **Pipelining** — splitting the datapath into 5 pipeline stages
- **Hazard detection** — handling data and control hazards
- **Forwarding** — resolving hazards without stalling
- **Branch prediction** — reducing control hazard penalty

## Project Background

Built as part of a 100-day public VLSI learning challenge, documenting daily progress on LinkedIn under **#100DaysOfVLSI**. Every module in this repository was designed, debugged, and verified individually before integration — no block was connected until its standalone behavior was confirmed correct in simulation.
